is NOT the filter.

### What a portfolio project CAN and CANNOT fix
- **CANNOT fix:** German level, years-of-experience floor, volume lotteries, visa. These are
  ~62% of rejections and no GitHub repo touches them. The lever there is **channel**
  (warm/cold-founder email, agencies) — not code.
- **CAN fix:** the **Full-Stack TS/Next.js gap** (real, recurring, costs both applications AND
  live interviews), and the **"undeniable depth" signal** in GraphRAG / eval-guardrails / MCP
  that the three warm-outreach founders (Interloom, Bayshore, Blockbrain) explicitly screen
  on. Projects here don't beat the board funnel — they **convert warm-channel replies into
  offers**, which is exactly where the funnel currently leaks.

### Single biggest unsolved barrier
**Interview-to-offer conversion (0 of 6).** It is downstream of everything: structural walls
cap the top of the funnel at ~2% interview rate, but even when a warm conversation opens,
nothing closes. Two project-addressable contributors feed it: (1) the **TS/Next.js gap** that
gets probed live and fails (Alago), and (2) **lack of a single deep, demo-able centerpiece**
that lets him say "I'd ship X in week 1" with proof instead of "the pattern transfers."
Projects can't fix STAR/Pramp delivery, but they remove the two technical gaps the delivery
currently has to apologize for.

## Project Ideas

Ranked. All bias toward the agentic-AI / GraphRAG / eval-guardrails / TS-Next.js
win-conditions for the warm-outreach startups.

### 1. GraphRAG Studio — full-stack agentic GraphRAG app (Next.js + Python)
- **Pitch:** A deployed web app where a user uploads docs, watches a typed knowledge graph
  build live, and chats over k-hop subgraph retrieval with citations.
- **Attacks:** The #1 lever — simultaneously kills the **TS/Next.js gap** (the Alago-class
  loss) AND makes the **GraphRAG centerpiece** undeniable for Interloom/Bayshore/Blockbrain.
- **Stack:** Next.js + TypeScript + React frontend; FastAPI + existing `graphrag-agent`
  (networkx) backend; LLM entity/relation extraction; Docker; Vercel (front) + small
  VPS/Render (API).
- **Scope:** Multi-week (flagship; build the frontend on top of the repo he already has).
- **Signal:** "Ships across the stack" + "deployed, not a notebook" + "graph retrieval is his
  real lane." Founders WILL click it.
- **Priority justification:** One project closes the most-probed gap and strengthens the
  most-screened strength. Highest EV.

### 2. RAG Reliability Harness — eval + guardrails layer
- **Pitch:** Extend `rag-eval-system` with Pydantic/Guardrails structured-output validation,
  LLM-as-judge scoring, and a production feedback loop with a metrics dashboard.
- **Attacks:** Bayshore's entire pitch is "deterministic guardrails, eval frameworks, feedback
  loops." Rarest skill he brings.
- **Stack:** Python, Pydantic, Guardrails-AI, RAGAs, LLM-as-judge, Langfuse, FastAPI.
  (Personal repo — never attribute to Perinet.)
- **Scope:** 1-week.
- **Signal:** "Thinks about reliability/production safety," not just demos.
- **Priority justification:** Most differentiated, least-common skill in the candidate pool;
  universal across all 3 warm targets.

### 3. MCP Server for the GraphRAG agent
- **Pitch:** Expose the knowledge-graph retrieval as an MCP server so any MCP client (Claude,
  Cursor) can query the graph as a tool.
- **Attacks:** Bayshore explicitly lists "MCP ecosystems, tool-calling architectures." Hot in
  mid-2026, cheap, differentiating.
- **Stack:** Python MCP SDK, FastAPI, existing graph backend.
- **Scope:** Weekend (1–2 days).
- **Signal:** "Current with the 2026 agent tooling frontier."
- **Priority justification:** Tiny cost, high topicality, plugs straight into #1.

### 4. Next.js + LLM mini-app (TS gap-closer, standalone)
- **Pitch:** Small Next.js app with an API route calling an LLM and a clean React UI — minimal
  artifact proving TS competence.
- **Attacks:** The **Full-Stack TS/Next.js trap** directly, as a hedge if #1 slips.
- **Stack:** Next.js, TypeScript, React, one LLM API route.
- **Scope:** Weekend.
- **Signal:** Lets him say "ready to pick up TS" with a repo behind it (honest framing, no
  faked depth).
- **Priority justification:** Build only if #1 stalls — otherwise #1 subsumes it. Insurance.

### 5. Agentic workflow with eval-in-the-loop (LangGraph + judge gating)
- **Pitch:** A multi-agent pipeline where an LLM-judge gates each step and bad outputs get
  retried/rerouted — agents + eval fused, not bolted on.
- **Attacks:** The agentic-AI core all 3 targets are built on; shows agents AND reliability
  together.
- **Stack:** LangGraph, FastAPI, LLM-as-judge, Langfuse, Docker. Builds on
  `multi-agent-pipeline`.
- **Scope:** 1-week.
- **Signal:** "Builds agentic systems that don't silently fail."
- **Priority justification:** Strong, but overlaps #2; value partly captured once #1+#2 exist.

### 6. Ontology-aware extraction module (graph depth-deepener)
- **Pitch:** Add a typed ontology + entity-resolution/dedup to the graph build so the KG is
  schema-correct, not just nodes-and-edges.
- **Attacks:** All 3 founders came up through knowledge graphs/ontologies; separates "did
  GraphRAG once" from "understands graph modeling."
- **Stack:** Python, networkx, Pydantic schema, ontology-constrained LLM extraction.
- **Scope:** 1-week.
- **Signal:** "Real graph-modeling depth."
- **Priority justification:** High-value for the exact warm targets but it's an enhancement to
  #1 — sequence as a #1 follow-on.

### 7. Live demo + README screenshots pass (signal-packaging, not new code)
- **Pitch:** Deploy #1, capture GIFs/screenshots, write founder-facing READMEs with
  architecture diagrams across the top 3 repos.
- **Attacks:** The funnel leak where founders click a repo and bounce off a bare README; makes
  warm-channel clicks convert.
- **Stack:** Vercel/Render deploy, README + Mermaid/Excalidraw diagrams.
- **Scope:** Weekend.
- **Signal:** "Production-minded, presents work like a senior." Memory flags repos must be
  public + clickable before outreach.
- **Priority justification:** Cheapest multiplier on every other project; do immediately after
  #1 is live.

### Recommended build order
**#1 (flagship) → #2 (eval/guardrails) → #3 (MCP) → #7 (packaging)**, with #6 as a #1
follow-on and #5 optional. #4 only as a hedge. Closes the live-probed TS gap and the
founder-screened depth gaps with one flagship plus three cheap satellites — then pair with
Pramp/STAR prep, since projects remove the *technical* apology in interviews but cannot fix
delivery on their own.

**Honest caveat:** projects address only the ~16% skill/lane slice plus interview-conversion
support. They do **not** move the ~62% structural rejections — that needs the channel pivot
(warm/cold-founder email + agencies), already identified in memory as the 15x-better lane.
Projects are the *conversion* engine for that channel, not a replacement for it.
